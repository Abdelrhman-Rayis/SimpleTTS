## Bugs

### B1 — pdfBlob stored as 0 bytes (P1) — FIXED

- **Where:** `index.html:2699`
- **Root cause:** PDF.js 3.11.174's `pdfjsLib.getDocument({ data: ab })` transfers ownership of the ArrayBuffer; after that call `ab.byteLength === 0`, so `new Blob([ab], { type: 'application/pdf' })` produces an empty Blob.
- **Impact:** Every doc added to the library was unreopenable. Clicking a library card after reload would error out at PDF.js with "The PDF file is empty, i.e. its size is zero bytes." The "if (!meta || !blob)" guard at line 1972 does not catch this case (a 0-byte Blob is truthy).
- **Repro:** Open any PDF → check IDB store `simpletts-library-v1 / pdfBlob` → `rec.blob.size === 0`.
- **Fix:** Construct the saved Blob from `file` (the original `File` is independent of the detached AB) instead of `ab`. One-line edit.
- **Verified:** After fix, fresh upload stores `blobBytes === fileSize` with valid `%PDF-` signature, reopen path loads the doc cleanly.

### B2 — Welcome/home view ignores dark theme (P2) — DOCUMENTED, NOT FIXED

- **Where:** `index.html:651-691` (rules inside `.viewer.home-mode`, `.hero-copy`, `.welcome-step`, `.hero-stat`, `.hero-spotlight`, `.hero-tag`, `.doc-card`, `.mix-card`, `.home-nav button`, `.home-search`, …)
- **Root cause:** Session 11 redesign hardcoded `#f7f3ea` / `#fffdf9` / `#1a1f2e` / `#b8924a` / `#62584e` directly instead of using `--bg` / `--paper` / `--text` / `--accent`.
- **Impact:** When user toggles dark theme on the welcome page, everything below the topbar stays cream-on-dark-text. Reader view is unaffected.
- **Why not fixed:** This is large-surface visual work (≈ 40 hardcoded colors over 40+ rules) and unclear whether Session 11 intended the welcome page to *always* use the warm light palette as a brand statement. Worth a UI design call before patching. Replace literal hex with the relevant CSS variables and add a `:root[data-theme="dark"]` override block.

### B3 — Kokoro unavailable in `.venv311` (config, not code)

- **Where:** Environment.
- **Root cause:** `.venv311` is missing `kokoro`, `soundfile`, `numpy`, `torch`. Detection in `kokoro_voice.py` is correct and reports unavailable.
- **Impact:** On `.venv311`, only Piper voices are usable.
- **Fix options:** (a) Switch the QA launch config to `/Users/rayis/OpenWebTTS/venv/bin/python` (already does in the original `SimpleTTS API Server` entry). (b) Install ML deps into `.venv311` (`pip install kokoro soundfile numpy torch`).

### B4 — `pageInput` element missing `max` attribute (cosmetic)

- **Where:** `index.html:975` — `<input type="number" id="pageInput" min="1" value="1" onchange="jumpToPage()" disabled>`
- **Impact:** Browser doesn't show the upper bound via native spinner / validation. JS `jumpToPage()` clamps with `Math.min(totalPages, val)` so behavior is correct.
- **Suggested fix:** After `/extract` resolves, set `pageInput.max = data.total_pages` alongside the `totalPages` text update.
